# 기능 구현 워크플로우

사용자가 요청한 기능을 3단계(조사 → 구현 → 리팩토링 검토)로 진행한다.

**요청 내용**: $ARGUMENTS

---

## 준비: 플랜 번호 결정

`.claude/plans/` 디렉토리에서 기존 파일의 번호를 확인하고, 다음 번호를 결정하라.
예) 기존에 `01-EXPLAIN.md`가 있으면 → 이번은 `02`번.

이하 `{NN}`은 결정된 번호(zero-padded 2자리)를 의미한다.

---

## Phase 1: 조사 및 계획 (EXPLAIN)

1. 요청 내용과 관련된 코드를 읽고 현재 상태를 파악하라.
2. 구현 내용이나 방향이 명확하지 않은 경우, EXPLAIN 문서를 작성하기 **전에** 사용자에게 먼저 질문하라.
   - 가능한 대안을 2~3개 제시하고, 각 장단점을 간략히 설명하라.
   - 사용자가 방향을 선택하거나 추가 요구사항을 명확히 한 뒤에 다음 단계로 진행하라.
3. 변경 방안을 조사하라. 근거(가이드라인, 수치, 비교)를 포함할 것.
4. `.claude/plans/{NN}-EXPLAIN.md`에 아래 구조로 작성하라:
   - **현재 상태**: 관련 파일, 코드, 문제점
   - **변경 사항**: 구체적인 diff 수준의 변경 내용 + 근거
5. 사용자에게 계획을 요약하고 진행 여부를 확인하라.

**이 단계에서 코드를 수정하지 마라.** 문서 작성만 한다.

---

## Phase 2: 구현 및 검증 (RESULT)

사용자가 승인하면 진행한다.

1. EXPLAIN에 작성한 계획대로 코드를 수정하라.
2. `/check` 실행 (lint + build). 실패 시 수정 후 재시도.
3. `.claude/plans/{NN}-RESULT.md`에 아래 구조로 작성하라:
   - **변경 파일**: 수정한 파일 목록
   - **변경 내역**: 각 변경의 before/after diff + 이유
   - **검증**: lint/build 통과 여부
4. 사용자에게 구현 결과를 요약하고, 현재 방향이 만족스러운지 확인하라.
   - 만족스러우면 Phase 3로 진행한다.
   - 만족스럽지 않으면, 사용자가 **구체적인 수정 방향**을 제시해야 한다. 제시된 방향에 따라 Phase 2를 다시 수행하라.

---

## Phase 3: 리팩토링 검토 (REFACTOR)

1. RESULT의 변경 사항을 바탕으로 리팩토링 후보를 검토하라.
   - 검토 관점: 레이아웃 정합성, prop 분리 필요성, 공통 컴포넌트 추출, 네이밍, 불필요한 추상화 여부
   - 각 후보에 대해 **현재 상태 → 대안 → 판단(필요/불필요) + 근거**를 기록
   - 각 후보에 **필요도**(🔴 높음 / 🟡 보통 / 🟢 낮음)를 함께 표기하고, 판단 이유를 명시하라.
2. `.claude/plans/{NN}-REFACTOR.md`에 작성하라.
3. 리팩토링이 필요한 항목이 있으면 사용자에게 진행 여부를 확인하라.

---

## 규칙

- Phase 간 전환 시 반드시 사용자 승인을 받아라.
- 각 Phase의 문서는 해당 Phase 시작 시 작성하라 (미리 작성 금지).
- 코드 수정은 Phase 2에서만 한다.
